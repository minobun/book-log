## メモ
### スタートアップはどこが違うのか。
スタートアップはイテレーションを重ねる。
プロダクトにものすごく集中する。


### スタートアップの関心事
顧客・インパクト・学習が重要視されていた。

- 存在が保証されていない。
- 生か死か。プロダクトの強さ次第。
- プロダクトマーケットフィットを求めて
  - プロダクトマーケットフィット：適切な市場に向けた最適なプロダクトを見つけ出すこと
  - 完璧に達成すると下記の状態になる。
    - プロダクトを作るスピードが追い付かない
    - サーバーを追加しても利用の拡大に追い付かない
    - 資金が流入しすぎて使いきれない
    - 人材需要に採用が追い付かない
- 大量の未知と向き合う

スタートアップはスピード以上に学習を重要視している。
どうすれば競合に打ち勝てるのか考えている。
イテレーションを重ね、開発者に対して権限を与える。

スタートアップやテック企業はプロダクトの開発力と顧客へのサービス提供能力で生死が決まる。


エンタープライズ企業では、システム開発の名目は生産性の向上と業務の効率化になる。
実際新規プロダクト開発の世界と社内業務システム開発の世界が衝突している。
エンタープライズ企業も新規プロダクトや新規サービスで対応しなければならないが、スピードとペースには慣れていない。


スタートアップは顧客にフォーカスしていて、市場にあるギャップの見極めが得意である。

エンタープライズ企業の過ち
1. プロダクト開発を社内業務システム開発と同じように扱ってしまう。
2. チームに十分な自律性と信頼を与えない。

エンタープライズ企業では計画通りに進めることが評価される。


| エンタープライズ | スタートアップ       |

ソフトウェアデリバリーは、これまでに思いつかなかった新規プロダクトや新規サービス、機能を「発見」するための手段である。

プロダクト開発は社内業務システム開発とは全く別のゲームである。



1. 成功を再定義する。
   1. 発見と学習

__ミッション__ を重要視する。

### ミッション
テック企業はプロジェクトではなく、ミッションで目的を与える。

プロジェクトの問題点
1. 期間が短い
2. フィードバックの機会が少ない
3. プロジェクトの融通が利かない
4. プロジェクトは力を奪う。
5. プロジェクトは間違ったことにフォーカスしている。
   1. 本当に意味があることは、顧客が欲しいと思うものを作ること。

ミッションは抽象度の高い目標
Spotifyの例としては下記のようなミッションが挙げられていた。
- 新しい音楽を簡単に見つけられるようにする。
- リビングルームを制する。
- 朝の通勤のお供になる。
ミッションはプロダクトではなくともよい。
- ほかのエンジニアリングチームがもっと速く進めるようにする。
- クラウドストレージの設定を簡単にする。
- 会社の株式公開準備を整える。
  
ミッションによるメリット
- チームの自発性を高める。
- 目的を意識させる。
- 仕事そのものにフォーカスさせる。

| プロジェクト                   | ミッション                       |
| ------------------------------ | -------------------------------- |
| 終わりがある                   | 期間に定めがない                 |
| 短期間                         | 長期間                           |
| プロジェクトマネージャーがいる | プロジェクトマネージャーがいない |
| 開発だけして引き継ぐ           | 開発もメンテナンスもする         |
| 完成したら解散する             | チームは一緒のまま               |
| 計画にフォーカス               | 顧客にフォーカス                 |
| 期待に応じることが価値         | インパクトが価値                 |
| トップダウン                   | ボトムアップ                     |

チームは自分たちで自分たちの仕事を生み出す。
何を作る必要があるのかを決めるのはチームである。

ユーザーストーリを一緒に作っていく。


### スクワッド
スクワッド：少人数の、自律した、必要な権限を持ったチーム @Spotify
デリバリーするプロダクトの隅々まですべてに責任を持つメンバーがそろったチーム
- 作ったものを自分たちでメンテナンスする。
- 自分たちで自分たちの仕事を生み出す。
→Henrik Knibergという方が作成している[Spotify Engineering Culture](https://www.youtube.com/watch?v=Yvfz4HGtoPc)素晴らしい解説動画がある。

エンタープライズ企業のアジャイルとは、権限と信頼が段違いであること。

学びはメンテナンスのなかにある。


スクワッドは計画よりもインパクトを重視する。
例として
- 新規登録数は？
- MAUは？
- リテンションは？
- NPSは？
- 音楽のトレンドは？
- リリースまでの速度は？

指標にフォーカスすることで余計な心配事を減らす。


スクワッドは最初のバージョンが正しくないと心得ている。
スクワッドは自律している。
スクワッドはは手を動かすメンバーだけで構成される。

プロダクトマネージャー：物事を最後まで見届けられる能力を備えた推進役
データサイエンティスト：データを使って意思決定をするように支援する。

Spotifyは巨大なモノリスだったが、小さなパーツに分割した。

経営リーダーのためのヒント
- チームに発奮興起してもらう。
- チームに間違えてもらう。
- 「間違え」に備えよう
- 楽しい名前を選んでもらおう。

### チーム編成

トライブ：担当するミッションが類似、関連しているスクワッドがまとまったもの→これ自体がミニ企業のような形
チャプター：同じ専門性を持つメンバーで構成される。
ギルド：同じ専門分野に興味のあるメンバーからなるグループ→公式なものではない。

トライブの原則
- スクワッド第一
- サーバントリーダー
- ミッションが肝心
- ミッション特化の古スタック編成
- 人数は重要
- トライブ内での異動はOK

意思決定をできるだけ現場に任せることが自己組織化につながる。
スクワッド→何を　にフォーカス
チャプター→どうやって　を手助けする

### Bet
2014年Spotifyは重要な物事を進められない状態にあった。
カンパニーベット：会社が取り組みたい重工事項を、終わらせたい順に並べたリスト
全社レベルでフォーカスを絞るために一度に取り組むことを少なくするというもの

DIBBという考え方
意思決定や議論に使われる
- Data：イテレーションの対象となるプロダクトやコンテンツ、ビジネスモデルらを示すデータ
- Insight：現実のデータから導き出された見解や、結論、または学んだこと
- Belief：チームに対して向かうべき方向を示す。
- Bet：実際にテストすることに決めた確信
